# 下一阶段开发计划

最后更新：2026-04-23

## 目标

把项目从“功能较全的家庭财务工作台”推进到“诊断驱动、任务闭环、结构可维护”的版本。

当前优先级不是继续加页面，而是先完成：

- 工程质量基线恢复
- 前端结构收敛
- 产品主流程重做
- 高密页面压缩与逻辑统一

## 当前诊断摘要

### 产品体验

- 首次进入没有明确起步路径，默认直接落到样例数据和完整工作台
- 总览页首屏过重，展示多于行动
- 诊断跳转后缺少目标定位和处理闭环
- 各业务页普遍把录入表单放在决策引导前面

### UI / 美学

- 视觉统一感主要停留在配色和圆角，没有形成稳定系统
- 头部导航、品牌区和总览首屏在争抢第一视觉层级
- 全站卡片偏多，页面节奏被切碎
- 字号层级不稳定，高密页面缺少渐进展开

### 前端实现

- `PlannerDataContext` 承担过多职责
- `DashboardPage`、`PortfolioPage` 等页面过重
- `App.css` 体量过大，缺少明确分层
- 当前只有 `build` / `lint`，且 lint 基线刚恢复，测试体系仍缺失

## Phase 1：质量基线与结构收敛

### 目标

- 恢复可持续开发的工程基础
- 为后续产品重做腾出安全空间

### 任务

1. 修复并稳定质量门禁
   - 保证 `npm run build` / `npm run lint` 稳定通过
   - 新增 CI 检查工作流，部署前先通过检查
   - 将部署流程从 `npm install` 切到 `npm ci`

2. 拆分上下文和页面职责
   - 把 `PlannerDataContext` 的上下文声明、Provider、hook 分离
   - 减轻页面文件职责，逐步把计算和视图拆开

3. 收敛样式体系
   - 把全局样式拆向 `tokens / layout / common / page`
   - 停止继续往单个 `App.css` 叠加大块新规则

4. 建立最小自动化测试
   - 优先覆盖诊断规则、预算计算、投资组合计算、导入导出校验

### 验收标准

- 本地 `build` / `lint` 可重复通过
- GitHub Actions 有独立质量检查流程
- 核心上下文和 hook 不再全部挤在一个文件中
- Phase 2 需要改的页面有明确拆分边界

## Phase 2：产品主流程重做

### 目标

把当前项目从“多模块展示”调整成“用户知道下一步做什么”的产品。

### 任务

1. 重做首次使用路径
   - 增加空状态
   - 增加数据完整度提示
   - 提供 3 步初始化清单

2. 重做总览页定位
   - 从宣传式首页收敛为任务首页
   - 首屏只保留核心结论、最高优先级问题、下一步动作

3. 打通诊断闭环
   - 诊断跳转带参数落到具体模块
   - 目标区域高亮、自动聚焦、预设筛选或预填
   - 支持处理后返回确认

4. 建立预警中心
   - 顶部铃铛不再只是红点
   - 用户可以查看问题、理解原因、直接处理

### 验收标准

- 新用户 3 分钟内能完成起步
- 诊断页每条高优先级问题都能直接落到具体操作
- 总览首屏动作优先级清楚

## Phase 3：高密页面与产品语言统一

### 目标

提升可信度、清晰度和长期维护性。

### 任务

1. 清理伪实时和占位型洞察
   - 总览页只保留可追溯的真实指标

2. 压缩高密页面
   - 投资组合页拆成 `总览 / 持仓 / 再平衡`
   - 数据配置页拆成 `基础配置 / 数据管理 / 历史审计`

3. 统一产品命名和逻辑口径
   - 修正导航命名
   - 让 `riskProfile` 等输入真正参与逻辑，或降低其承诺

### 验收标准

- 高密页面结构更易扫描
- 同类模块标题、卡片、字段语义一致
- 所有建议都能说明数据来源
